# Visual Regression Tests

Pixel-level screenshot comparison tests that detect unintended visual
changes to the Agent Deck web app (the redesigned Preact shell under
`internal/web/static/app/`). A run fails if any screenshot differs from
its committed baseline by more than 0.1% of pixels
(maxDiffPixelRatio: 0.001) or 200 absolute pixels.

The suite runs weekly via `.github/workflows/weekly-regression.yml`
against a real server with all API endpoints mocked to deterministic
fixtures (see `visual-helpers.ts`).

## Baselines are CI-generated (IMPORTANT)

Baselines committed to `__screenshots__/` MUST be generated by the CI
runner, not on a developer machine. Font rasterization, GPU/compositor
behavior, and Chrome builds differ across environments; locally
generated baselines fail in CI and vice versa.

### Regenerating baselines

```bash
# 1. Push your branch, then trigger the workflow in baseline mode:
gh workflow run weekly-regression.yml --ref <branch> -f update_baselines=true

# 2. Wait for the run to finish, then download the artifact:
gh run list --workflow=weekly-regression.yml --branch <branch> --limit 1   # grab <run-id>
gh run download <run-id> --name visual-baselines-<run-id> \
  --dir tests/e2e/visual-regression/__screenshots__/

# 3. Review the PNGs, then force-add and commit
#    (.git/info/exclude has a *.png rule, hence -f):
git add -f tests/e2e/visual-regression/__screenshots__/
```

`update_baselines=true` runs Playwright with `--update-snapshots`,
uploads the regenerated `__screenshots__/` tree as the
`visual-baselines-<run_id>` artifact, and skips Lighthouse and issue
creation. It never commits or pushes anything itself.

There is also a `skip_issue=true` input that runs the normal checks but
suppresses the auto-filed regression issue (useful while iterating on
the suite itself).

### PR requirements for baseline changes

When your PR updates baseline screenshots, you MUST:

1. Describe which baselines changed and why in the PR description
2. Confirm the visual change is intentional
3. Note the workflow run ID the baselines came from

Reviewers treat unexplained baseline changes as a signal that a PR may
have introduced a visual regression that was silently accepted.

## Running tests locally (debugging only)

Local runs are for debugging locator failures and eyeballing diffs.
**Never commit locally generated baselines** — they will not match the
CI environment.

Use the pinned Playwright Docker image so your local rendering is at
least close to CI:

```bash
# 1. Build the binaries
make build
go build -o ./build/agent-deck-test-server ./cmd/agent-deck-test-server

# 2. Start a test server (agent-deck-test-server is the TUI-less entry
#    point; it accepts -listen and -profile)
AGENTDECK_PROFILE=_test ./build/agent-deck-test-server -listen 127.0.0.1:18420 &

# Wait for readiness
for i in $(seq 1 15); do
  curl -sf http://127.0.0.1:18420/healthz && break
  sleep 1
done

# 3. Run the suite in Docker
docker run --rm --network=host \
  -v $(pwd)/tests/e2e:/work \
  -w /work \
  -e PLAYWRIGHT_BASE_URL='http://127.0.0.1:18420/?token=test' \
  mcr.microsoft.com/playwright:v1.59.1-jammy \
  npx playwright test --config=pw-visual-regression.config.ts
```

To debug a single test, add `-g "test name"`. To see what the new UI
actually renders, add `--update-snapshots` and inspect the generated
PNGs — then **delete them before committing**:

```bash
git checkout -- tests/e2e/visual-regression/__screenshots__/ 2>/dev/null
git clean -fd tests/e2e/visual-regression/__screenshots__/
```

The Docker image tag `v1.59.1-jammy` is pinned to match the
`@playwright/test` version in `tests/e2e/package.json`. Do not upgrade
one without the other.

## Adding new tests

1. Create or extend a `.spec.ts` file in `tests/e2e/visual-regression/`
2. Import helpers from `./visual-helpers.js`
3. Call `freezeClock(page)` and `mockEndpoints(page)` BEFORE `page.goto()`
4. Call `prepareForScreenshot(page)` after page load (and again after any
   interaction that changes the view)
5. Pass `getDynamicContentMasks(page)` as the `mask` option; add
   `chartMasks(page)` for any view containing Chart.js canvases
6. Ground every selector in the component source under
   `internal/web/static/app/` — never guess selectors
7. Regenerate baselines via the CI workflow (see above), download the
   artifact, and commit spec + baseline together

### Example test

```typescript
import { test, expect } from '@playwright/test';
import {
  freezeClock, mockEndpoints, prepareForScreenshot,
  getDynamicContentMasks,
} from './visual-helpers.js';

test.describe('My feature baselines', () => {
  test('my feature at 1280x800', async ({ page }) => {
    await freezeClock(page);
    await mockEndpoints(page);
    await page.goto('/?token=test');
    await prepareForScreenshot(page);
    const masks = await getDynamicContentMasks(page);
    await expect(page).toHaveScreenshot('my-feature-1280x800.png', { mask: masks });
  });
});
```

## Thresholds

| Parameter | Value | Rationale |
|-----------|-------|-----------|
| maxDiffPixelRatio | 0.001 | 0.1% of pixels; catches layout shifts while allowing sub-pixel anti-aliasing variance |
| maxDiffPixels | 200 | Absolute cap; even on a 1920x1080 screenshot (2M pixels), 200 changed pixels is a meaningful regression |
| threshold | 0.2 | Per-pixel color distance; 0.2 on [0,1] scale allows minor sub-pixel color shifts |

## Determinism strategy

- `freezeClock()` pins `Date`/timers to 2026-01-01T00:00:00Z
- `mockEndpoints()` fixtures every API endpoint the shell calls
  (`/api/menu`, `/api/costs/*`, `/api/profiles`, `/api/settings`,
  `/api/system/stats`) and aborts the `/events/menu` SSE stream so the
  connection pill settles on a deterministic "disconnected"
- `prepareForScreenshot()` kills CSS animations, waits for web fonts,
  the profile dropdown, and the settled connection state
- Chart.js canvases are masked (canvas anti-aliasing is not bit-stable);
  the xterm terminal frame interior is masked in the terminal-fill test
  (its content depends on a live tmux session)

## Troubleshooting

**Tests pass locally but fail in CI (or vice versa):**
Your baselines were probably generated locally. Regenerate them via the
CI workflow (see "Regenerating baselines").

**False positive on a specific element:**
Add the element's selector to `getDynamicContentMasks()` in
`visual-helpers.ts` so it gets masked in all screenshots — but first ask
whether the element can be made deterministic via a fixture instead.

**Baseline PNGs not tracked by git:**
The `.git/info/exclude` file contains `*.png` which blocks PNGs from
git tracking. Use `git add -f` to force-add baseline PNGs.

**Docker image not available locally:**
```bash
docker pull mcr.microsoft.com/playwright:v1.59.1-jammy
```
